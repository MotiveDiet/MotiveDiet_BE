## 운영 전후 검증

배포 전에 프로덕션 DB에서 다음을 확인해야 합니다.

```sql
SHOW CREATE TABLE food_category;

SELECT
    @@character_set_server,
    @@character_set_database,
    @@character_set_connection,
    @@character_set_client,
    @@character_set_results,
    @@collation_connection;
```

필수 조건:

- `name`, `emoji`의 실제 charset이 `utf8mb4`
- JDBC 연결의 `character_set_client`와 `character_set_connection`이 `utf8mb4`
- 특히 `emoji`가 `utf8`/`utf8mb3`가 아니어야 함

DDL 정본에는 이미 [schema-changes.sql](/Users/seok_hwan_/MotiveDiet/MotiveDiet/MotiveDiet_BE/schema-changes.sql:24)처럼 `DEFAULT CHARSET=utf8mb4`가 지정돼 있습니다. 하지만 수동 DDL 방식이므로 실제 prod 테이블도 별도로 확인해야 합니다.

현재 손상을 바이트 단위로 기록합니다.

```sql
SELECT
    id,
    name,
    HEX(name) AS name_hex,
    emoji,
    HEX(emoji) AS emoji_hex,
    weekly_threshold
FROM food_category
ORDER BY id;
```

U+FFFD는 UTF-8에서 `EFBFBD`입니다. 배포 후에는 결과가 정확히 다음이어야 합니다.

| id | name HEX | emoji HEX | threshold |
|---:|---|---|---:|
| 1 | `ECB998ED82A8` | `F09F8D97` | 2 |
| 2 | `ED9684EBB284EAB1B0` | `F09F8D94` | 3 |
| 3 | `EB9DBCEBA9B4` | `F09F8D9C` | 2 |
| 4 | `EBB9B5` | `F09FA590` | 3 |
| 5 | `EC88A0` | `F09F8DBA` | 2 |
| 6 | `EC95BCE식` 오류 없이 `EC95BCECB식`이 아니라 정확한 DB `HEX(name)`을 원문과 대조 | `F09F8C99` | 2 |

마지막 행처럼 사람이 HEX를 손으로 옮기면 실수하기 쉬우므로, 가장 확실한 최종 판정은 API 응답이 정확히 `야식`, `🌙`인지 확인하고 DB에서 아래 조건이 0인지 확인하는 것입니다.

```sql
SELECT COUNT(*) AS replacement_character_count
FROM food_category
WHERE HEX(name) LIKE '%EFBFBD%'
   OR HEX(emoji) LIKE '%EFBFBD%';
```

그리고 정확한 정본 비교:

```sql
SELECT id, name, emoji, weekly_threshold
FROM food_category
WHERE NOT (
       (id = 1 AND name = '치킨'   AND emoji = '🍗' AND weekly_threshold = 2)
    OR (id = 2 AND name = '햄버거' AND emoji = '🍔' AND weekly_threshold = 3)
    OR (id = 3 AND name = '라면'   AND emoji = '🍜' AND weekly_threshold = 2)
    OR (id = 4 AND name = '빵'     AND emoji = '🥐' AND weekly_threshold = 3)
    OR (id = 5 AND name = '술'     AND emoji = '🍺' AND weekly_threshold = 2)
    OR (id = 6 AND name = '야식'   AND emoji = '🌙' AND weekly_threshold = 2)
);
```

결과가 0행이고 총 행 수가 6이면 시드 범위는 완전히 복구된 것입니다.

`DB_URL`에 `characterEncoding=UTF-8`을 hotfix로 추가할 필요는 없습니다. Connector/J가 서버와 `utf8mb4`로 정상 협상한다면 중복 설정입니다. 위 세션 조회에서 연결 charset이 잘못된 경우에만 Railway의 `DB_URL`을 별도 교정해야 하며, 그 경우에도 yml에 실제 URL이나 기본값을 넣으면 안 됩니다.
