# 코딩 컨벤션

## 단순함이 먼저다

- 요청받지 않은 기능·추상화·설정 가능성을 넣지 않는다
- 구현체가 하나뿐인 인터페이스, 제품이 하나뿐인 팩토리, 절대 안 변하는 값의 config → 만들지 않는다
- 200줄을 썼는데 50줄로 되면 다시 쓴다

## 변경은 외과적으로

- 요청과 직접 연결되지 않는 줄은 건드리지 않는다. 인접 코드·주석·포맷을 "개선"하지 않는다
- 기존 스타일을 따른다. 내 취향과 달라도 맞춘다
- 내 변경이 고아로 만든 import·변수·함수는 내가 지운다
- 기존 데드코드는 **언급만 하고 지우지 않는다**

## 단순화하면 안 되는 것

신뢰 경계의 입력 검증 · 데이터 유실을 막는 에러 처리 · 보안 조치 · 명시적으로 요청된 것

## 주석

- 코드가 스스로 말하는 것을 주석으로 반복하지 않는다
- 코드로 표현할 수 없는 **제약**만 적는다 (왜 이 값인지, 무엇이 깨지는지)
- "여기서 X를 한다" 같은 리뷰어용 설명은 머지되는 순간 소음이 된다

## 검증

비자명한 로직(분기·루프·파서·인증/보안 경로)에는 실행 가능한 체크를 하나 남긴다.
`BUILD SUCCESSFUL`만으로 통과시키지 마라 — 테스트가 스킵돼도 초록불이다.
`build/test-results/test/*.xml`에서 `tests="N" failures="0"`을 확인할 것.
