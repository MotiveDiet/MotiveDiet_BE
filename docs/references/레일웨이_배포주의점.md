# Railway 배포 주의점

MotiveDiet_BE를 Railway에 배포하면서 겪은 에러와 해결 방법 기록. 에러 생길 때마다 여기 추가.

---

## 1. Java 버전 불일치

- **증상**: 빌드가 Gradle 다운로드까지는 진행되다 실패
  ```
  Cannot find a Java installation on your machine ... matching: {languageVersion=17, ...}
  ```
- **원인**: `build.gradle`에 `JavaLanguageVersion.of(17)` (Java 17 toolchain) 지정돼 있는데, Railway의 빌더(Railpack)는 기본값인 Java 21만 설치함. Railpack이 Gradle toolchain 블록은 자동 감지 못 함
- **해결**: 레포 루트에 `.java-version` 파일 생성, 내용은 `17` 한 줄. Railpack이 이 파일을 읽어서 Java 17을 설치함



## 2. 단일 모듈 Gradle 프로젝트의 배포 시작 커맨드 오류

- **증상**: 빌드는 성공하는데 배포 시작 시 크래시
  ```
  ls: cannot access '*/build/libs/*jar': No such file or directory
  ```
- **원인**: Railpack이 자동 생성한 시작 커맨드가 `*/build/libs/*.jar` (하위 모듈 디렉토리 안의 build/libs를 가정)를 찾는데, 이 프로젝트는 단일 모듈이라 jar가 바로 `build/libs/*.jar`에 있음. 멀티모듈 Gradle 프로젝트를 가정한 Railpack의 기본 휴리스틱이 안 맞음
- **해결**: 레포 루트에 `railpack.json` 생성해서 시작 커맨드 직접 지정
  ```json
  {
    "deploy": {
      "startCommand": "java $JAVA_OPTS -jar -Dserver.port=$PORT $(ls -1 build/libs/*jar | grep -v plain)"
    }
  }
  ```



## 3. 환경변수 미설정 + DB 미연결

- **증상**: 배포는 되는데 앱이 계속 크래시
  ```
  The following 1 profile is active: "local"
  ...
  Unable to determine Dialect without JDBC metadata (please set 'jakarta.persistence.jdbc.url' ...)
  ```
- **원인**: Railway 서비스에 `SPRING_PROFILES_ACTIVE`를 설정 안 해서 기본값 `local` 프로필로 뜸. 게다가 MySQL DB 자체가 Railway 프로젝트에 없어서 `DB_URL` 등 접속 정보도 없었음
- **해결**:
  1. `railway add --database mysql`로 MySQL 플러그인 추가
  2. `MotiveDiet_BE` 서비스에 아래 환경변수 설정 (Railway 대시보드 or `railway variables --set`)
    - `SPRING_PROFILES_ACTIVE=prod`
    - `DB_URL` = `jdbc:mysql://${{MySQL.MYSQLHOST}}:${{MySQL.MYSQLPORT}}/${{MySQL.MYSQLDATABASE}}` (Railway 서비스 간 참조 변수 문법, MySQL 서비스 이름에 맞게)
    - `DB_USER_NAME` = `${{MySQL.MYSQLUSER}}`
    - `DB_PASSWORD` = `${{MySQL.MYSQLPASSWORD}}`
    - `JWT_SECRET`, `ACCESS_TOKEN_VALIDITY_IN_MILLISECONDS`, `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `GOOGLE_REDIRECT_URI` — 실제 값은 절대 이 파일이나 git에 커밋하지 말고 Railway 환경변수로만 관리



## 4. `ddl-auto: validate`는 스키마가 미리 있어야 함

- **증상**: DB는 연결되는데 여전히 크래시 (Dialect/JDBC 에러 또는 테이블 없음 에러)
- **원인**: `application-prod.yml`은 `hibernate.ddl-auto: validate`라서 Hibernate가 테이블을 만들어주지 않고 "이미 존재하는 스키마와 엔티티가 일치하는지"만 검사함. Flyway/Liquibase 같은 마이그레이션 도구가 없는 상태라 빈 DB에 배포하면 검증할 테이블 자체가 없어서 실패
- **해결**: 엔티티 기반으로 `CREATE TABLE` SQL을 직접 작성해서 배포 전에 Railway MySQL에 미리 적용 (`mysql -h <public-proxy-host> -P <port> -u root -p<password> railway < schema.sql`)
- **⚠️ 앞으로 주의**: 엔티티가 추가/변경될 때마다 이 방식으로는 스키마를 수동으로 맞춰줘야 함. 엔티티가 늘어나면 Flyway/Liquibase 도입을 검토할 것

## 기타 참고

- Railway 배포 관련 CLI/MCP: `use-railway` 스킬 + Railway MCP 서버가 로컬에 설치돼 있음 (`railway setup agent`로 설치)
- 로그 확인: `railway logs --build --latest` (빌드 로그), `railway logs --deployment --latest` (실행 로그)
- 배포 상태: `railway status`
- `railway` 명령어가 안 먹히면 Homebrew가 PATH에 없는 것. 아래 한 번만 실행하면 이후 모든 터미널에서 계속 사용 가능:
  ```bash
  echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> ~/.zprofile
  ```
  (실행 후 터미널 새로 열거나 `source ~/.zprofile`)
- `railway status`, `railway logs` 등은 프로젝트가 연결된 디렉토리(`MotiveDiet_BE`)에서 실행해야 함
