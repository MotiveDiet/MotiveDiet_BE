# PRD: 동기 기반 팩폭 다이어트 코칭 앱 (MotiveDiet)

**타겟 플랫폼**: iOS 우선

---

## 1. Problem Statement

다이어트를 시작하는 사용자 다수는 목표를 막연하게 세우거나, 세우더라도 동기가 시간이 지나며 흐려져 중도 포기한다. 기존 식단 기록 앱은 사진 인식 기반 로깅 정확도에 집중하지만 지속적인 동기부여 기능은 약하고, 코칭 톤을 다양화한 앱도 이미 존재한다. 그러나 사용자가 입력한 "개인적 동기"(예: 특정 이벤트, 이유)를 실시간 코칭 메시지 내용에 직접 엮어주는 서비스는 시장에 확인되지 않았다. 이 격차를 방치하면 사용자는 일반적인 팩폭/응원 메시지에 곧 무뎌지고, 기존 다이어트 앱과 동일한 이탈

---

## 3. Non-Goals

1. **정밀 칼로리/영양 계산** — 로깅 마찰 최소화가 우선이므로 이번 버전은 대략적 추정만 (정밀도는 별도 초기)
2. **사람 코치 결합 하이브리드 모델** — 효과는 검증됐으나(미시간대 연구: AI+휴먼이 AI 단독 대비 74% 더 높은 감량 효과) 운영 복잡도가 커 v2 이후 검토
3. **Android 출시** — iOS 우선 출시, Android는 iOS 학습 이후 별도 스펙
4. **팩폭/음식 유머 메시지 생성 모델 파인튜닝** — MVP는 기존 LLM에 구조화 신호(동기 이벤트, 음식 빈도 횟수) + 프롬프트로 생성(5·6절). 파인튜닝은 톤 일관성 확보 목적으로 구현 완료 후 별도 연구 단계에서 검토 (콘텐츠 가이드라인은 프롬프트 레이어에서 즉시 통제 가능해야 하므로 우선)

---



## 4. User Stories



### 신규 사용자

- 신규 사용자로서, 목표 체중·기한·다이어트 동기를 입력한다. 팩폭 코칭이 내 상황에 맞게 개인화되도록 하기 위해서다.



### 재방문 사용자

- 재방문 사용자로서, 즐겨찾기한 음식을 원탭으로 기록한다. 매 끼니 입력 부담을 줄이기 위해서다.
- 재방문 사용자로서, 살찌는 음식을 먹었을 때 내 동기와 연결된 팩폭 메시지를 받는다. 목표를 계속 의식하게 하기 위해서다.
- 재방문 사용자로서, 특정 음식을 반복해서 먹었을 때 빈도 기반 유머 메시지를 받는다. 반복되는 식습관을 재미있게 자각하게 하기 위해서다.
- 사용자로서, 언제든 팩폭 강도를 낮추거나 끌 수 있다. 코칭 방식에 대한 통제권을 갖기 위해서다.

### Acceptance Criteria (예시)

**동기 파싱 파이프라인**

- 사용자가 온보딩에서 동기를 자유텍스트로 입력했을 때
- 시스템이 텍스트를 파싱하면
- 구조화 신호만 DB에 저장되고 원문은 저장소에 남지 않아야 한다
- 원문 처리 방식 결정: 원문 문자열 자체는 저장하지 않되, 파싱 시점에 팩폭에 필요한 뉘앙스(이벤트 유형, 대상, 날짜, 원문에서 뽑은 1~2문장 paraphrase)까지 구조화 필드에 함께 저장한다. paraphrase는 원문을 그대로 재구성할 수 없는 수준으로 요약해 "원문 폐기" 요건은 지키면서 팩폭 품질 저하는 방지한다 (예: `motive_type=기념일`, `target=여자친구`, `paraphrase="여자친구 생일 전까지 살 빼기"`)

## 6. 음식 빈도 기반 펀치라인 팩폭 레이어

기존 "동기 기반 팩폭"(사용자가 입력한 동기-이벤트를 참조하는 개인화 메시지)과는 별도 개념의 레이어이지만, 실제 구현은 5절의 동기 연동 팩폭과 **같은 LLM 호출 한 번**을 공유한다 (7-6절 결정). 특정 음식을 일정 빈도 이상 반복 섭취했을 때 이 호출의 프롬프트에 빈도 컨텍스트가 추가되고, 동기 신호와도 결합해 콤보 메시지를 생성할 수 있다.

**결정(2026-07-14)**: tier1/2/3 같은 등급 구분을 없애고, **FoodCategory마다 주간 임계값을 숫자 하나로** 지정한다 (예: 치킨 2회, 햄버거 3회, 라면 2회). 이 임계값 이상 섭취한 로그부터는 매번(2번째든 5번째든) **GPT-5 API를 실시간 호출**하면서 "이번 주 이 음식을 몇 번째 먹었는지" 실제 횟수를 프롬프트에 그대로 알려준다 — 숫자가 매번 다르므로 GPT가 만드는 문장도 자연히 매번 달라지고, "몇 번째까지 어떤 문구"를 사람이 미리 설계할 필요가 없다. 별도의 추가 호출이 아니라 5절 팩폭 생성과 같은 호출이다.

예:

- 2번째: "치킨 2번째네. 벌써 이러기야?"
- 4번째(동기 신호가 D-14 이내로 유효할 때): "치킨 벌써 4번째... 너 이러다 닭 되겠어. 여자친구 생일 D-12인데 이럴 거야?"



### 6.1 데이터 모델


| 엔티티          | 설명                                                                    |
| ------------ | --------------------------------------------------------------------- |
| FoodCategory | 음식 카테고리 (치킨, 라면, 빵, 술, 야식 등) + 카테고리별 주간 임계값 숫자 하나(`weeklyThreshold`) |
| UserFoodLog  | 사용자별 최근 7일 로그, 카테고리별 카운트 산출용                                          |

별도의 `FrequencyThreshold`/`PunchlineTemplate` 테이블은 두지 않는다 — 임계값은 `FoodCategory`에 컬럼 하나로 충분하고, 문구는 매번 실시간 생성이라 저장해둘 풀 자체가 없다.



### 6.2 동작 규칙

- 이 메시지는 잠금화면 알림에는 노출하지 않고 앱 내부에서만 노출한다 (기존 잠금화면 정책과 동일)
- **프롬프트 조합 규칙** (음식 로그 1건당 GPT-5 호출은 항상 1회): (1) 음식명은 항상 포함 (2) 유효한 동기 신호가 있으면 `target`/`paraphrase`는 항상 포함해 일반적인 동기 참조를 하되, `eventDate` 기반 D-day 카운트다운("D-12" 같은 구체적 숫자)은 D-14 이내일 때만 포함한다 (3) 이번 주 해당 카테고리 누적 횟수가 `FoodCategory.weeklyThreshold` 이상이면, 임계값에 처음 도달했을 때뿐 아니라 **그 이후 같은 주에 같은 카테고리를 로깅할 때마다 매번** 실제 누적 횟수를 포함한다. 이 세 조건의 조합 결과에 따라 메시지가 자연히 "단독 팩폭 / 동기 참조 팩폭 / 빈도 펀치라인 / 동기+빈도 콤보"로 갈린다



### 6.3 콘텐츠 가이드라인

- 문구를 DB에 저장해두고 콘텐츠팀이 수정하는 방식이 아니므로, "콘텐츠 톤 관리"는 GPT-5 호출 시스템 프롬프트(톤 규칙, 금지 표현)를 수정하는 방식으로 대체한다
- 5절 팩폭 생성과 동일한 콘텐츠 가이드라인(7-3절 (a)~(d): 인신공격 금지, 자해/섭식장애 언급 금지, 차별 표현 금지, 욕설 금지)을 시스템 프롬프트에 고정 반영한다
- 반복되는 문구가 거슬릴 경우(선택 사항) 최근 생성된 문구 몇 개를 프롬프트에 "피해야 할 표현"으로 포함하는 방식을 고려할 수 있으나, MVP 범위에는 포함하지 않는다

## 7. Decisions

각 결정이 어느 개발 단계에 걸리는지는 `ROADMAP.md`에 정리되어 있음. 1인 개발 단계이므로 법무/데이터 검증이 필요한 항목도 우선 결정을 내리고, 재검토 트리거가 오면 다시 연다.

**1. 살찌는 음식 판정 기준 — 룰 기반 채택**

- 결정: MVP는 룰 기반. 6.1절 `FoodCategory` 테이블에 고칼로리/고당/고지방/야식 태그를 미리 매핑해 판정 (치킨/피자/라면/빵/술/야식 등). 별도 분류기 없이 카테고리 매핑만으로 처리
- 근거: MVP 단계엔 분류기 학습用 로그 데이터가 없고, 판정 기준이 결정적(deterministic)이어야 팩폭 트리거를 예측 가능하게 유지 가능
- 재검토 트리거: 로그 데이터가 충분히 쌓이고 카테고리 매핑으로 커버 안 되는 음식(비정형 입력 등)이 많아지면 분류기 도입 검토 (v2)

**2. 안전 가드레일 자동 판정 — 폐기 (2026-07-16), 도입하지 않음**

- 결정: 입력 텍스트에서 위험 신호를 자동 판정해 안전 톤으로 전환하는 설계 전체를 폐기한다. 기존 설계(deny-list 1차 필터 + LLM 분류기 2차 판정 + confidence 임계치 폴백)는 구현하지 않는다
- 근거:
  - (판정 불가) confidence 임계치를 정할 근거 데이터도, 검증할 수단도 MVP 단계엔 없다. 튜닝할 수 없는 임계치는 안전을 보장하지 못하면서 구현 비용만 발생시킨다
  - (오탐 비용) 한국어 다이어트 앱 특성상 동기 입력란에 "살 안 빠져 죽겠어" 같은 일상 과장 표현이 **정상 입력으로** 들어온다. 키워드로 recall을 노리면 핵심 유저층이 통째로 오탐된다
  - (입력면 협소) 자유텍스트 입력 지점은 온보딩 동기 1곳·1회뿐이고, 이후 로깅은 전부 원탭이라 판정할 트래픽 자체가 거의 없다
- 대체 수단: 팩폭 콘텐츠의 안전은 **3번 콘텐츠 가이드라인 (a)~(d)를 시스템 프롬프트에 고정 반영**하는 방식으로만 확보한다(6.3절). 사용자 통제 수단(opt-in 동의, 강도 조절/끄기)은 5절 그대로 유지
- 재검토 트리거: App Store 심사에서 Guideline 1.1 관련 반려 사유가 나오거나, WoZ 테스트에서 실제 위험 사례가 관측되면 다시 연다

**3. 팩폭 문구 콘텐츠 가이드라인 승인자 — 프로덕트 오너(본인) 1차 승인**

- 결정: 현재 1인 개발 단계이므로 프로덕트 오너가 아래 가이드라인을 작성·승인하고 즉시 적용
  - (a) 외모·체중·능력에 대한 인신공격 금지, 행동(식습관)에 대한 유머만 허용
  - (b) 자해·자살·섭식장애 관련 언급 절대 금지 (2번 폐기 이후, 이 항목이 해당 소재를 배제하는 유일한 수단)
  - (c) 특정 집단 비하·차별적 표현 금지
  - (d) 욕설·혐오 표현 금지
- 근거: 소규모 팀에서 정식 법무 프로세스를 기다리는 것은 비현실적. 명확한 룰을 문서화해 즉시 적용 가능하게 함
- 재검토 트리거: 팀 확장 또는 외부 투자/법무 리소스 확보 시 정식 법무 검토로 전환


**6. LLM 벤더 및 호출 구조 — GPT-5로 통일, 음식 로그 1건당 호출 1회**

- 결정: 동기 파싱처럼 가볍고 빠른 판단은 `gpt-5-mini`, 팩폭·펀치라인처럼 톤이 중요한 생성은 `gpt-5`로 벤더를 GPT-5 계열 하나로 통일한다. 또한 5절 "동기 연동 팩폭"과 6절 "음식 빈도 펀치라인"은 별개 API 호출이 아니라, 음식 로그 1건당 `gpt-5` 호출 1회로 통합한다 (프롬프트 조합 규칙은 6.2절 참고)
- 근거: (벤더 통일) API 키·클라이언트를 하나만 관리하면 되어 운영이 단순해짐. (호출 통합) 분리 호출 시 임계값을 넘는 로그에서 팩폭 메시지와 빈도 메시지가 각각 따로 응답에 실려 사용자에게 메시지 2개가 뜨는 문제가 있었음 — 하나로 합쳐야 PRD 6절 예시("치킨 벌써 4번째... 닭 되겠어. 여자친구 생일 D-12인데...")처럼 자연스러운 한 문장이 나옴
- 재검토 트리거: 동기 파싱의 구조화 품질이나 팩폭 생성의 톤 품질이 GPT-5 단일 벤더로 부족하다고 판단되면 해당 기능만 다른 벤더로 분리 검토
