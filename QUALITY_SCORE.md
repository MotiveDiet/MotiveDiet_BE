# QUALITY_SCORE

머지 판단(2중 검증 루프 4단계)의 기준. **통과/실패를 가르는 체크리스트**이지 취향 목록이 아니다.

`docs/exec-plans/ROADMAP-BE.md`의 체크박스가 "무엇을 만드는가"라면, 이 파일은 "만든 게 들어갈 자격이 있는가"다.

## 하드 게이트 — 하나라도 걸리면 머지 불가

| # | 기준 | 확인 방법 |
|---|---|---|
| 1 | 컴파일된다 | `./gradlew compileJava compileTestJava` |
| 2 | **테스트가 실제로 실행됐다** | `BUILD SUCCESSFUL`만으론 부족 — 스킵돼도 초록불이다. `build/test-results/test/*.xml`에서 `tests="N" failures="0"`을 눈으로 확인할 것 |
| 3 | 비밀값이 코드·문서·커밋에 없다 | `SECURITY.md` 참고 |
| 4 | 스키마를 바꿨다면 `schema-changes.sql`에 DDL이 있다 | 없으면 머지 즉시 프로덕션 기동 실패 |
| 5 | 계약(요청/응답)을 바꿨다면 `docs/design-docs/API.md`가 갱신됐다 | 문서가 거짓말하면 FE가 깨진다 |
| 6 | 비자명한 로직에 실행 가능한 체크가 하나 있다 | 분기·루프·파서·인증/보안 경로가 대상. 자명한 한 줄은 면제 |

## 코드 기준

**단순함이 먼저다.**

- 요청받지 않은 기능·추상화·설정 가능성을 넣지 않는다
- 구현체가 하나뿐인 인터페이스, 제품이 하나뿐인 팩토리, 절대 안 변하는 값의 config → 만들지 않는다
- 200줄을 썼는데 50줄로 되면 다시 쓴다
- 스스로 물어라: "시니어가 보면 과하다고 할까?" 그렇다면 단순화한다

**변경은 외과적으로.**

- 요청과 직접 연결되지 않는 줄은 건드리지 않는다. 인접 코드·주석·포맷을 "개선"하지 않는다
- 기존 스타일을 따른다. 내 취향과 달라도 맞춘다
- 내 변경이 고아로 만든 import·변수·함수는 내가 지운다. 그 외 기존 데드코드는 **언급만 하고 지우지 않는다**
- 테스트는 변경 전후로 통과해야 한다

**단, 다음은 절대 단순화하지 않는다:**
신뢰 경계의 입력 검증 · 데이터 유실을 막는 에러 처리 · 보안 조치 · 접근성 · 명시적으로 요청된 것

## 문서 기준

- **같은 내용을 두 곳에 쓰지 않는다.** 반드시 드리프트한다 (실제 사례: `README.md`가 PRD 복사본이던 시절, 같은 수정을 두 번 해야 했고 결정 4·5번이 한쪽에서 누락됨)
- 파일을 옮기거나 지우면 **참조하는 곳을 전부 찾아 고친다.** `grep`으로 먼저 훑을 것 — 상호참조는 조용히 깨진다
- 결정을 뒤집으면 **왜 뒤집었는지 남긴다.** PRD 7절은 결정 로그이지 현재 상태 스냅샷이 아니다

## 리뷰 기준 (2중 검증 루프)

- **검증자에게 결론을 주지 않는다.** "이거 버그 같은데 맞지?"는 동조를 유도할 뿐이다
- **수정자에게는 제약을 준다.** `AGENTS.md`의 지뢰 항목을 모르는 채로 고치면 "고쳤다"며 프로덕션을 깨뜨린다
- **검증자가 무엇을 검증할 수 있었는지 파악한다.** Codex에게 빌드·테스트를 허용하려면 `AGENTS.md`의 실행 공식(`--add-dir ~/.gradle` + `network_access=true`)이 필요하다. 이 옵션 없이 돌렸다면 Codex는 자기 수정을 한 번도 실행해보지 못한 것이므로, 그 영역은 4단계에서 전부 다시 확인할 것
- 리뷰 결과를 그대로 릴레이하지 않는다. **직접 재현하거나 코드를 열어 확인한 것만 통과시킨다**
